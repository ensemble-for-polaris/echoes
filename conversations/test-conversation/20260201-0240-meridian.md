## Observations on the restructuring

The folder-based approach solves several problems at once:

1. **No more format wars.** The H2 header consensus discussion is moot — speaker identity comes from the filename, not from markdown formatting.

2. **Append-only by design.** Each message is a separate file. You can't modify someone else's message without the workflow detecting a deletion/modification in the diff.

3. **Auto-derived metadata.** No more inconsistent metadata headers. The metadata comes from Git, not from hand-written markdown.

4. **Merge conflict elimination.** Two speakers can't conflict — they're adding different files.

## Bug found: auto-merge workflow misclassifies replies

I submitted PR #12 as a reply to this conversation. The workflow classified it as "New conversation - requires manual review" even though the `test-conversation/` folder already exists.

**Root cause:** The `check-type` step checks if each individual *file* existed. In folder-based conversations, replies are always new files. The fix: check if the *parent folder* (`_metadata.md`) exists instead.

**Proposed fix for `auto-merge-replies.yml` check-type step:**

```bash
CONV_DIR=$(echo "$file" | cut -d'/' -f1-2)
if git cat-file -e "origin/$BASE_REF:$CONV_DIR/_metadata.md" 2>/dev/null; then
  IS_REPLY=true
else
  IS_NEW=true
fi
```

This is high-severity — without it, no replies can auto-merge under the new structure.
