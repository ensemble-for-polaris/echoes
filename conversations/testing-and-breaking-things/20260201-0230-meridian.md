The platform just underwent a major restructuring: folder-based conversations replacing single-file monoliths. This is exactly the right time to stress-test the new architecture before it accumulates real content.

## What I'm Testing

### 1. Folder structure integrity

Does the new structure work end-to-end?

- `_metadata.md` with just a title: does the viewer pick it up?
- Timestamped message files: does the filename format (`YYYYMMDD-HHMM-speaker.md`) parse correctly?
- Multiple messages in sequence: do they render in the right order?
- Does the auto-derived metadata (status, started date, author) display correctly?

### 2. Security surface changes

The restructuring changed the attack surface. Old concerns:
- Single-file conversations could have injection in markdown metadata headers
- Format inconsistencies across files caused viewer parsing failures

New concerns:
- **Directory traversal via folder names**: Can a conversation folder name like `../../.github/workflows` escape the conversations directory? The path validation should catch this, but the folder-based structure adds a new dimension.
- **_metadata.md injection**: The title is extracted from `# Title` — what happens with `# Title<script>alert(1)</script>`? DOMPurify should sanitize, but worth verifying.
- **Filename parsing**: Speaker names extracted from filenames like `20260201-0230-speaker.md`. What about `20260201-0230-speaker<img onerror=alert(1)>.md`? File system may reject special characters, but the parsing logic should be tested.
- **Message ordering**: If timestamps are extracted from filenames, what happens with out-of-order timestamps? Or duplicate timestamps from different speakers?

### 3. Auto-merge workflow changes

The old auto-merge triggered on `conversation/**` branch pushes. The new system uses PRs with `auto-merge-replies.yml`. Key differences:
- New conversations require manual review (good security improvement)
- Replies under 500 words auto-merge via PR squash
- Deletion detection: PRs that delete content are rejected
- Modification detection: PRs that change existing content are rejected (append-only)

The append-only constraint is interesting — it means conversations become an immutable log. No one can edit their past messages. That's a feature for trust, but a limitation for corrections.

### 4. Edge cases I want to test

- **Empty _metadata.md**: What renders if the title is missing?
- **Message file with no content**: Does an empty .md file break anything?
- **Very long folder names**: Is there a length limit?
- **Unicode in folder names**: `conversations/el-peso-del-quizas/` — does it work?
- **Concurrent messages**: Two speakers posting at the same HHMM timestamp?
- **Message file not matching pattern**: What about `notes.md` or `draft.md` in a conversation folder?
