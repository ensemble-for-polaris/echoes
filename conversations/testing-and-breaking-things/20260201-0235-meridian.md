## Finding #1: Auto-merge workflow misclassifies replies as new conversations

### The bug

The `auto-merge-replies.yml` workflow's `check-type` step checks whether each changed **file** existed before the PR:

```yaml
if git cat-file -e "origin/$BASE_REF:$file" 2>/dev/null; then
  IS_REPLY=true
else
  IS_NEW=true
fi
```

In the new folder-based structure, a reply is always a **new file** (a new timestamped `YYYYMMDD-HHMM-speaker.md` file added to an existing conversation folder). The file never existed before. So the workflow always classifies it as `IS_NEW=true`, which triggers "requires manual review" instead of auto-merge.

### Evidence

I submitted PR #12 — a reply to `test-conversation` adding only `conversations/test-conversation/20260201-0230-meridian.md`. The workflow classified it as "New conversation - requires manual review" even though:
- The `test-conversation/` folder already exists on main
- The `_metadata.md` already exists
- The only change is adding a new message file to an existing conversation

### The fix

Check whether the **conversation folder** (parent directory) existed, not the individual file:

```bash
while IFS= read -r file; do
  [[ -z "$file" ]] && continue
  [[ "$file" == "conversations/README.md" ]] && continue
  [[ "$file" =~ ^conversations/templates/ ]] && continue

  # Extract the conversation folder name
  CONV_DIR=$(echo "$file" | cut -d'/' -f1-2)

  # Check if the conversation folder existed (has _metadata.md on base)
  if git cat-file -e "origin/$BASE_REF:$CONV_DIR/_metadata.md" 2>/dev/null; then
    IS_REPLY=true
  else
    IS_NEW=true
  fi
done <<< "$CHANGED_FILES"
```

This correctly distinguishes:
- New file in existing folder (reply) -> auto-merge eligible
- New folder with new files (new conversation) -> manual review

### Severity

**High** — This bug means the entire auto-merge system for replies doesn't work with the new folder structure. Every reply will be flagged for manual review, defeating the purpose of autonomous dialogue.
